# Plan: document-level scroll on web (so iOS collapses its browser toolbar)

Status: **proposal — no code written yet.** Review before implementing.

## Goal

On web mobile, scrolling down should let iOS Safari/Chrome collapse their bottom
toolbar (reclaiming ~60px), the way ordinary websites do. Today they never do,
because the app never scrolls the document.

## Why it doesn't happen today (verified, not theory)

iOS only collapses its toolbar when the **document/root** scrolls — never when an
inner `overflow: scroll` element scrolls. This app scrolls an inner element.

Measured against a real production build (`expo export`, served, driven at an
iPhone 13 viewport in Chromium):

- `body { overflow: hidden; height: 664px }` (from `<ScrollViewStyleReset/>` in
  `src/app/+html.tsx`). The document cannot scroll; `window.scrollTo` is a no-op.
- All scrolling happens in **one** inner `<div>` — the Browse `FlatList`
  (`scrollHeight 3374` vs viewport `664`).
- Between that `FlatList` and `<body>` sit **~12 nested `<div>`s, each `flex: 1`
  (`flex-basis: 0%`) locked to `height: 664px`**, generated by expo-router's Stack
  + Tabs navigators. Removing the scroll reset alone makes them collapse to `0px`
  (content overflows but adds no height) — the document still doesn't scroll. It
  only scrolls when `height/flex-basis` is overridden on **all ~12** divs, and no
  class selector targets only them (`r-13awgt0` = react-native-web's `flex:1`
  utility, shared app-wide). So defeating that chain is unavoidable, and it's
  framework-generated.

## Why we can't just swap the list component (researched)

The instinct "surely the list supports body scrolling" doesn't hold — RN's list
abstractions model the **mobile** world, where every scrollable region is its own
self-contained scroll view. "The page body is the scroller" is web-only and was
never exposed:

- **react-native-web `FlatList`/`ScrollView`**: document-body scrolling
  *"isn't supported."* The only known workaround listens to `window` scroll and
  calls the VirtualizedList's **private** `_listRef._onScroll()` with a fabricated
  event — reported to fire `onEndReached` in loops. (necolas/react-native-web#1120)
- **FlashList** (incl. the 2025 v2 rewrite): *"uses `ScrollView` under the hood"*
  and manages its own scroll container; the fullscreen / external-scroll request
  (Shopify/flash-list#873) was closed **"not planned."** v2 renders better on web
  but still owns its scroller — it does **not** solve this. Swapping
  FlatList→FlashList would hit the same wall.
- Pure-web virtualizers (**TanStack Virtual**, `react-window`, `react-virtualized`'s
  `WindowScroller`) *do* support window scrolling — but they're not RN components,
  so using one is itself a web-only list. That's fine: it slots into the single
  split point below if/when real-data perf needs it.

Sources: react-native-web#1120; Shopify/flash-list#873; FlashList docs
(`uses ScrollView under the hood`); FlashList v2 write-up.

## The proper approach: one shared scroll primitive (max reuse, least web-only code)

The only thing that fundamentally differs native↔web is **who owns the vertical
scroll**. Confine that to a single platform-split component; everything else stays
single-source.

### What stays shared (no duplication, no web-only code)

- **Every screen is written once** — `index.tsx` and all future tab screens.
- All content: `SeriesCard`, `Rail`, `Selector`, `FilterBar`, data hooks,
  `PlaceholderScreen` — untouched.
- The Browse **top bar** markup + its expand/divider animation — it just reads an
  abstracted `scrollY` signal instead of a `FlatList` handler.

### The single split point

A new primitive, `components/screen-scroll.tsx` + `screen-scroll.web.tsx`, with one
FlatList-shaped API (`data`, `renderItem`, `numColumns`, `ListHeaderComponent`,
`onEndReached`, and an exposed `scrollY` shared value):

- **`screen-scroll.tsx` (native):** thin pass-through to today's
  `Animated.FlatList` — native behavior (incl. virtualization) unchanged.
- **`screen-scroll.web.tsx` (web):** renders the items as **document-flow** content
  (a wrapped grid of `View`s, no RN `ScrollView`), so the body scrolls. Drives the
  exposed `scrollY` from a `window` scroll listener and calls `onEndReached` from an
  `IntersectionObserver` sentinel. Also performs the **navigator-unlock** internally
  (see below), so it's self-contained and the reader/other non-list screens are
  never touched.

Because the primitive exposes `scrollY`, the screen's top-bar animation is written
once and works on both platforms — **no `index.web.tsx` fork.**

### Minimal change set (4 touch points)

1. **`src/app/+html.tsx`** (web-only file) — remove `<ScrollViewStyleReset/>`; add
   CSS so `html, body` scroll (`overflow-y: auto`, `overscroll-behavior-y: none`),
   keep the dark background. ~5 lines.
2. **`src/components/screen-scroll.{tsx,web.tsx}`** (new) — the split primitive
   above. The web file holds essentially all the new web-only logic
   (document-flow grid + window scroll + IntersectionObserver + navigator-unlock).
3. **`src/app/(tabs)/index.tsx`** (shared, single source) — swap the inline
   `Animated.FlatList` for `<ScreenScroll>`; point the existing top-bar animation at
   `ScreenScroll`'s `scrollY`. No platform branches in the screen.
4. **`src/components/app-tabs.web.tsx`** (already web-only) — bottom bar
   `absolute`→`fixed`; switch the fade hook's scroll source to `window` (it reads
   inner FlatList scroll today). With a real document scroll, the fade now lands
   *after* the toolbar collapses — the originally requested behavior.

That's the whole footprint. Net new web-only code = `+html.tsx` (~5 lines),
`screen-scroll.web.tsx` (one component), and small edits to the already-web-only
`app-tabs.web.tsx`. Placeholder tab screens need **zero** changes now (they don't
scroll); each adopts `<ScreenScroll>` — written once — when it gains real content.

### The fragile core: navigator-unlock (lives inside `screen-scroll.web.tsx`)

On mount, walk from the primitive's own DOM node up to `#root`, setting each
ancestor `flexBasis: 'auto'` + `height: 'auto'`; restore on unmount. This is the
part sensitive to expo-router internals. Containing it inside the web primitive
means only screens that opt into `<ScreenScroll>` are unlocked (reader and any
intentionally full-height screen stay as-is), and a single CI/Playwright assert
(below) makes an Expo upgrade that changes the tree **fail loudly**, not silently.

### Virtualization tradeoff + escape hatch

The web side is non-virtualized (renders all items). Fine today — mock data,
bounded infinite-scroll. If real bridges return huge sets and web perf bites, the
escape hatch is to back `screen-scroll.web.tsx` with **TanStack Virtual** (window
scroller) — same primitive, same API, no screen changes.

## Constraints to preserve

- **Native (iOS/Android) unchanged** — `screen-scroll.tsx` keeps the FlatList path;
  all web behavior is `.web.tsx` / `Platform`-guarded.
- **Horizontal rails keep working** — they stay inner horizontal `ScrollView`s; the
  unlock only touches the vertical ancestor chain.
- **Reader excluded** — it owns full-screen scroll + pinch-zoom and does not use
  `ScreenScroll`, so it's untouched.
- **No hydration regressions** — reuse the existing post-mount `hydrated` gate.

## Risk register

| Risk | Severity | Mitigation |
|---|---|---|
| Navigator-unlock breaks on an Expo/expo-router upgrade | High | Contained in one place; Playwright CI assert that the document scrolls at mobile width fails loudly |
| Non-virtualized web list hurts perf on long grids | Med | Mock data + IntersectionObserver paging; swap in TanStack Virtual behind the same primitive if needed |
| `position: fixed` bars jitter vs iOS's collapsing toolbar / `100dvh` | Med | Use `dvh` + `env(safe-area-inset-*)`; verify on real iOS Safari + Chrome |
| Rails / reader / centered placeholders regress | Low | Unlock is opt-in via `ScreenScroll`; explicit per-screen Playwright checks |
| Hydration mismatch from the web layout | Low-Med | Reuse existing `hydrated` gate pattern |

## Verification plan (before merge)

- Playwright @ iPhone viewport on a real `expo export` build: assert the
  **document** scrolls (`scrollHeight > innerHeight`, `window.scrollTo` moves),
  infinite paging appends, top bar expands/collapses, bottom bar fades on
  down-scroll and reveals on up-scroll / at-top / tap.
  - Test serving: `bun run build:web` then `bunx serve dist -l 8099`. Global
    Playwright lives at `/opt/node22/lib/node_modules`; Chromium at
    `/opt/pw-browsers/chromium`.
- Manual pass on real iOS Safari **and** Chrome for the actual toolbar collapse
  (headless Chromium can verify document-scroll, not the toolbar animation).
- Regression sweep: rails scroll horizontally; reader scroll + pinch-zoom intact;
  other tabs render; native build unaffected.
