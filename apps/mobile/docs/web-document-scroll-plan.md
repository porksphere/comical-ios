# Plan: document-level scroll on web (so iOS collapses its browser toolbar)

Status: **proposal — no code written yet.** Review before implementing.

## Goal

On web mobile, scrolling down should let iOS Safari/Chrome collapse their bottom
toolbar (reclaiming ~60px), the way ordinary websites do. Today they never do,
because the app never scrolls the document.

## Why it doesn't happen today (verified, not theory)

iOS only collapses its toolbar when the **document/root** scrolls — never when an
inner `overflow: scroll` element scrolls. This app scrolls an inner element.

Measured against a real production build (`expo export`, served, driven at an
iPhone 13 viewport in Chromium):

- `body { overflow: hidden; height: 664px }` (from `<ScrollViewStyleReset/>` in
  `src/app/+html.tsx`). The document cannot scroll (`scrollHeight == innerHeight`,
  `window.scrollTo` is a no-op).
- All scrolling happens in **one** inner `<div>` — the Browse `FlatList`
  (`scrollHeight 3374` vs viewport `664`).

### The real blocker (this is why it's not a one-line fix)

The file comment says "Remove `<ScrollViewStyleReset/>` for a global,
document-level scroll." That alone is **not sufficient** — proven empirically:

- Between the `FlatList` and `<body>` there are **~12 nested `<div>`s, each
  `flex: 1` (`flex-basis: 0%`) and locked to `height: 664px`**, generated by
  expo-router's Stack + Tabs navigators. They are not stylable from app code and
  carry build-hashed class names (`r-13awgt0` = react-native-web's `flex:1`
  utility — shared by *every* flex view in the app: cards, rows, rails — so there
  is **no class selector** that hits only the chain).
- Removing the reset and making `#root` content-height makes the chain **collapse
  to `0px`** (flex-basis:0 with no parent height to stretch into → zero-height
  boxes; content overflows visibly but contributes no layout height). Document
  still doesn't scroll.
- The document only scrolls when `height/flex-basis` is overridden on **every one
  of those ~12 generated divs** (brute-forced in a probe → `scrollHeight 3374`,
  `window.scrollTo(500)` works). There is no subset that suffices.

**Conclusion:** any solution must defeat the navigator's flex-height chain, and
that chain is framework-generated. There is no clean, declarative lever.

## Constraints to preserve

- **Native (iOS/Android) unchanged** — native has no document scroll and needs the
  `FlatList`. All changes are web-only (`.web.tsx` splits / `Platform.OS` guards).
- **Horizontal rails keep working** — they are inner `ScrollView`s; a blanket
  "kill inner overflow" rule would break them.
- **The reader is excluded** — it owns a full-screen scroll + pinch-zoom
  (`touch-action: none`, scoped listeners); it must keep its own model.
- **Existing scroll-driven UI keeps working** — the top-bar expand animation and
  the bottom-bar fade both read `FlatList` `onScroll` today; on web they'd have to
  read `window` scroll instead.
- **No hydration regressions** — the static export prerenders mobile; first client
  render must match (the app already handles this with post-mount `hydrated`
  flags).

## Options considered

### Option A — Stop (status quo). Recommended baseline.
Keep inner scroll. The bottom-bar fade already works (it reads inner scroll). Cost
0, risk 0; you just don't get the toolbar collapse. Listed for comparison.

### Option B — Pure CSS override. Rejected (not viable).
No selector can target the ~12 generated divs without also hitting every `flex:1`
view in the app. A universal `#root *` override breaks all layout. Dead end.

### Option C — Web document-scroll layout (recommended if we proceed).
The only approach that both works and stays contained. Five parts:

1. **`src/app/+html.tsx`** — remove `<ScrollViewStyleReset/>`; add CSS so
   `html, body` scroll (`overflow-y: auto`, `overscroll-behavior-y: none`) and keep
   the dark background. (Document-level only on web; native unaffected.)

2. **Unlock the navigator chain (the fragile core).** A web-only effect that, while
   a document-scroll route is mounted, walks from the screen container up to
   `#root` and sets each ancestor `flexBasis: 'auto'` + `height: 'auto'`, and
   restores on unmount. Scoped to the routes we opt in (Browse, and later the other
   tab screens) so the reader and any intentionally full-height screen are
   untouched. This is the part that is **fragile to expo-router internals** and the
   main maintenance risk; it needs a guard + a fast unit/Playwright check so a
   future Expo upgrade that changes the tree fails loudly, not silently.

3. **`src/app/(tabs)/index.web.tsx`** (new split) — render the Browse grid as
   **document-flow** content (no outer `ScrollView`/`FlatList`; a plain wrapped grid
   of `View`s). Keep the existing native `index.tsx` as-is. Reimplement:
   - infinite "Browse all" via an `IntersectionObserver` sentinel (replacing
     `onEndReached`),
   - the top-bar expand animation + divider off **`window` scroll** (replacing the
     reanimated `useAnimatedScrollHandler`).
   Horizontal rails inside stay as-is (inner horizontal `ScrollView`s are fine).

4. **Sticky bars → `position: fixed`.** The Browse top bar and the Tabs bottom bar
   become `fixed` to the viewport (today they're `absolute` over a non-scrolling
   parent). The bottom-bar fade hook switches its scroll source to `window`. With a
   real document scroll, the fade finally lands *after* the toolbar collapses — the
   originally requested behavior.

5. **Other tab screens (library/history/activity/settings).** Currently centered
   placeholders. They sit in the same unlocked chain, so verify they still center
   and don't collapse; opt each into the document-scroll path as it gains real
   content. Detail/series are pushed Stack screens — verify or exclude per route.

### Option D — Restructure to host content outside the navigator. Rejected.
Rendering the scroller outside expo-router's tree (portal into `body`) escapes the
flex chain but fights routing/transitions app-wide. Highest effort, highest risk.

## Recommended path

**Option C, rolled out Browse-first.** Implement parts 1–4 behind a web-mobile
guard, ship it for Browse only, verify, then extend to the other tab screens
(part 5). The reader stays on its own model throughout.

## Risk register

| Risk | Severity | Mitigation |
|---|---|---|
| Ancestor-unlock effect breaks on an Expo/expo-router upgrade | High | Scope to opted-in routes; add a Playwright assert (`document` scrolls at mobile width) to CI so breakage is loud, not silent |
| Losing `FlatList` virtualization on web hurts perf on long grids | Med | Mock data + `IntersectionObserver` paging; cap rendered pages; revisit only if real bridges return huge sets |
| Horizontal rails / reader / centered placeholders regress | Med | Route-scoped unlock (never global); explicit Playwright checks per screen |
| `position: fixed` bars jitter against iOS's collapsing toolbar / `100dvh` | Med | Use `dvh`/`env(safe-area-inset-*)`; test on real iOS Safari + Chrome |
| Hydration mismatch from the new web layout | Med | Reuse the existing post-mount `hydrated` gate pattern |
| Maintaining a second Browse implementation (`index.web.tsx`) | Low-Med | Keep shared pieces (cards, rails, data hooks) factored; only the scroll shell diverges |

## Verification plan (before merge)

- Playwright @ iPhone viewport on a real `expo export` build: assert the
  **document** scrolls (`scrollHeight > innerHeight`, `window.scrollTo` moves),
  infinite paging appends, top bar expands/collapses, bottom bar fades on
  down-scroll and reveals on up-scroll/at-top/tap.
- Manual pass on real iOS Safari **and** Chrome: confirm the toolbar actually
  collapses (headless Chromium can verify document-scroll, **not** the toolbar
  animation itself — that needs a device).
- Regression sweep: rails scroll horizontally; reader scroll + pinch-zoom intact;
  other tabs render; native build unaffected (`Platform` guards).

## Rough effort

~1–2 focused days for Browse-first (parts 1–4 + verification), plus smaller
follow-ups to extend to the remaining tab screens. The ancestor-unlock effect is
the part most likely to need iteration on real devices.
